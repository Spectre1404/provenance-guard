# Provenance Guard — Planning & Specification

> Written **before** implementation. This is the canonical spec that drives all
> code generation in Milestones 3–5. Updated before starting each stretch feature.

## Overview

Provenance Guard is a backend system that any creative-sharing platform can plug
into to assess whether submitted **text** was written by a human or generated by
AI. It does **not** police creativity — it protects attribution by:

1. Classifying submitted content with **multiple independent signals**.
2. Producing a **calibrated confidence score** that communicates uncertainty
   instead of forcing a binary verdict.
3. Surfacing a **plain-language transparency label** to readers.
4. Giving creators an **appeals path** when they believe they were misclassified.
5. Recording every decision in a **structured audit log**, behind **rate limiting**.

A core design principle: **a false positive (calling a human's work AI) is worse
than a false negative.** The scoring thresholds and label copy are deliberately
biased to avoid wrongly accusing human writers.

## Architecture

```
SUBMISSION FLOW
  Creator → POST /submit {text, creator_id}
     │ (rate limiter: 429 if over limit)
     ▼
  ┌─────────────┐   ┌──────────────┐
  │ Signal 1    │   │ Signal 2     │
  │ Groq LLM    │   │ Stylometry   │
  │ (semantic)  │   │ (structural) │
  └──────┬──────┘   └──────┬───────┘
   ai_prob 0-1        ai_prob 0-1
         └────────┬────────┘
                  ▼  combined = 0.65*LLM + 0.35*stylometry
          ┌───────────────┐
          │ Confidence    │  → ai_probability (0-1)
          │ Scorer        │  → attribution bucket
          └──────┬────────┘     (likely_human / uncertain / likely_ai)
                 ▼
          ┌───────────────┐
          │ Label Gen     │  → 1 of 3 transparency label variants
          └──────┬────────┘
                 ▼
          ┌───────────────┐      ┌───────────────────────────┐
          │ Audit Log     │─────▶│ Response JSON             │
          │ (SQLite)      │      │ {content_id, attribution, │
          └───────────────┘      │  confidence, signals,     │
                                 │  label}                   │
                                 └───────────────────────────┘

APPEAL FLOW
  Creator → POST /appeal {content_id, creator_reasoning}
                 │
                 ▼  look up original decision by content_id
          update content.status → "under_review"
                 ▼
          append appeal entry to Audit Log (original decision + reasoning)
                 ▼
          Response {content_id, status: "under_review", message}
```

**Narrative.** In the submission flow, raw text passes the rate limiter, then is
analyzed independently by two signals (semantic LLM + structural stylometry);
their `ai_probability` scores are combined into one calibrated confidence, mapped
to an attribution bucket and a transparency label, logged, and returned with a
`content_id`. In the appeal flow, a creator references that `content_id`, the
content's status flips to `under_review`, the appeal is logged alongside the
original decision, and a confirmation is returned — a human reviewer can later
read the original signals plus the creator's reasoning from the audit log.

### Project Structure (modules)

The code is split into small modules so each implementation milestone is a
drop-in addition rather than a rewrite of one large file.

```
provenance-guard/
├── app.py              # Flask app + all routes (/submit, /log, later /appeal …)
├── db.py               # SQLite store + structured audit log helpers
├── signals/
│   ├── __init__.py
│   ├── llm.py          # Signal 1 — Groq LLM classification (semantic)
│   ├── stylometry.py   # Signal 2 — stylometric heuristics  (added in M4)
│   └── repetition.py   # Signal 3 — n-gram/repetition detector (stretch)
├── scoring.py          # Confidence combination + bucket mapping (added in M4)
├── labels.py           # Transparency label generation (added in M5)
├── requirements.txt
├── planning.md
└── README.md
```

**Module responsibilities:**

- **`db.py`** — owns all persistence via SQLite (`provenance.db`). Two tables:
  `content` (one row per submission: current `status`, attribution, confidence)
  and `audit_log` (append-only structured events). Public helpers: `init_db`,
  `save_content`, `get_content`, `update_status`, `write_log`, `get_log`. The
  audit log is append-only by design — an appeal adds a new event rather than
  overwriting the original classification.
- **`signals/llm.py`** — Signal 1. `score_llm(text)` calls
  `llama-3.3-70b-versatile` via Groq with a JSON-only system prompt and returns
  `{"ai_probability": float, "rationale": str}`. Uses `temperature=0` and
  `response_format=json_object` for stable, parseable output; on any API/parse
  error it **degrades gracefully** to a neutral `0.5` ("uncertain") instead of
  crashing the request. Runnable standalone (`python -m signals.llm`) for
  inspection before wiring into the endpoint.
- **`app.py`** — the Flask application and route handlers. Generates the
  `content_id` (UUID4), produces an ISO-8601 UTC timestamp, calls the signal(s),
  persists to `db`, and returns the structured JSON response. In M3 the
  confidence is a placeholder (raw LLM score) and the label is a placeholder
  string; these are replaced by `scoring.py` (M4) and `labels.py` (M5).

> **M3 placeholders (intentional):** until M4/M5 land, `/submit` returns the raw
> LLM score as `confidence` and a literal placeholder `label` string. This lets
> the endpoint, persistence, and audit log be verified end-to-end with one signal
> before scoring and labels are built on top.

## 1. Detection Signals

The system uses **two genuinely independent signals**. "Independent" means they
measure different *kinds* of properties — one semantic, one structural — so their
agreement is more informative than either alone.

### Signal 1 — Groq LLM classification (semantic)

- **What it measures:** Whether the text *reads* as machine-written when judged
  holistically — coherence, hedging patterns, "safe"/generic phrasing, tonal
  evenness. Implemented by prompting `llama-3.3-70b-versatile` to return a
  structured JSON assessment.
- **Output:** `{ai_probability: float 0–1, rationale: str}`. The model is
  instructed to return a probability, not a yes/no, and to lean toward
  uncertainty (mid-range) when unsure.
- **Why it differs human vs AI:** LLM-generated prose tends to be fluent,
  evenly hedged, and structurally "tidy"; human writing carries idiosyncrasy,
  abrupt shifts, opinion, and noise the model can recognize.
- **Blind spot:** Non-deterministic and sometimes confidently wrong. Can be
  fooled by lightly edited AI text. **Biased against formal or non-native
  English writers** whose careful prose resembles AI output. We never let the
  LLM alone push someone into the "AI" bucket — it is weighted, and the
  threshold for "AI" is deliberately high.

### Signal 2 — Stylometric heuristics (structural)

Pure-Python statistics, no external libraries. We compute **six metrics** across
three families (rhythm, vocabulary, voice), normalize each to a 0–1
"AI-likeness" sub-score against rough human/AI reference ranges, then combine via
a weighted average into a single structural `ai_probability`. The reference
ranges are **heuristic, not trained** — they encode tendencies, not a learned
model.

| # | Metric (family) | What it captures | Direction | Weight |
|---|---|---|---|---|
| 1 | **Sentence-length burstiness** — coeff. of variation of words/sentence (rhythm) | Humans mix short & long sentences; AI is even | low CV → AI-like | 0.22 |
| 2 | **Type-token ratio** — unique/total words (vocabulary) | Vocabulary diversity | mid-range (~0.6) → AI-like | 0.12 |
| 3 | **Punctuation variety** — count of *distinct* mark types used (vocabulary) | Humans punctuate richly (– … ; : ( ) ! ?); AI leans comma/period | low variety → AI-like | 0.12 |
| 4 | **AI-marker lexical density** — rate of overused AI words per 100w (*furthermore, delve, crucial, "it is important to note", tapestry, leverage, seamless, realm, underscore…*) (voice) | The AI "register" | high → AI-like | 0.24 |
| 5 | **Human-voice markers** — rate of contractions + casual fillers per 100w (*honestly, kinda, ok so, like, i mean…*) (voice) | Casual human idiosyncrasy | high → human-like (lowers AI score) | 0.20 |
| 6 | **Sentence-opener diversity** — unique first-words ÷ sentences (rhythm) | AI starts sentences formulaically | low → AI-like | 0.10 |

- **Output:** `{ai_probability: float 0–1, metrics: {burstiness_cv,
  type_token_ratio, punct_variety, ai_markers_per_100w, human_markers_per_100w,
  opener_diversity, sentence_count, short_text_damped}}`.
- **Why it differs human vs AI:** AI optimizes for fluent uniformity and a
  characteristic register; humans are statistically irregular and idiosyncratic.
  Metrics 4 & 5 (lexical tells) and 1 (rhythm) are the strongest discriminators
  and carry the most weight.
- **Short-text damping:** with fewer than 3 sentences the rhythm/diversity stats
  are unreliable, so the combined stylometry score is pulled proportionally
  toward 0.5 (uncertain) rather than trusting a noisy statistic.
- **Blind spot:** **Completely blind to meaning.** A deliberately uniform human
  (legal, academic, technical) scores AI-like; a chaotic AI prompt-hack could
  score human-like; the AI-marker list can be gamed by avoiding those words.
  This is why stylometry gets the lower weight (0.35) and never decides alone.

### Combination

```
combined_ai_probability = 0.65 * llm.ai_probability + 0.35 * stylometry.ai_probability
```

The LLM is the stronger semantic signal and gets the majority weight; stylometry
is noisier (especially on short text) so it gets less, but still moves the score
and surfaces disagreement between the two.

## 2. Uncertainty Representation

The system **never returns a bare binary**. It returns a continuous
`ai_probability` ∈ [0,1] plus a derived attribution bucket and a user-facing
confidence level.

### What a score means

- `ai_probability` is the system's estimated probability that the text is
  AI-generated.
- **0.6 means:** "leaning AI, but genuinely unsure" — it lands in the *Uncertain*
  band, NOT an accusation. We intentionally make the score-to-user mapping
  conservative: a 0.6 must not read to a creator like a verdict.

### Thresholds (3 buckets)

| Combined `ai_probability` | Attribution bucket   | User-facing confidence            |
|---------------------------|----------------------|-----------------------------------|
| `0.00 – 0.40`             | `likely_human`       | High                              |
| `0.40 – 0.75`             | `uncertain`          | Low                               |
| `0.75 – 1.00`             | `likely_ai`          | High                              |

**Asymmetry (false-positive protection):** the "AI" bucket requires a high 0.75
to trigger, while "human" only needs < 0.40. The *Uncertain* band is wide
(0.40–0.75) on purpose — when in doubt, we say so rather than accuse.

### User-facing confidence level

Confidence reflects **distance from the uncertain zone**, not the raw score:

- In `likely_human` or `likely_ai`: **High** confidence (we're far from the
  ambiguous middle).
- In `uncertain`: **Low** confidence (signals are mixed / near the boundary).

This is what makes 0.51 vs 0.95 feel different to a reader: 0.51 → Uncertain/Low,
0.95 → Likely AI/High.

### How we validate scores are meaningful (M4)

We run ≥4 deliberately chosen inputs (clearly AI, clearly human, formal-human
borderline, lightly-edited-AI borderline) and confirm: clearly-AI scores high,
clearly-human scores low, borderline cases land in the Uncertain band, and the
two signals' individual scores are printed to diagnose disagreement.

## 3. Transparency Label Design

Three variants, plain language, written for a non-technical reader. Each makes
clear this is an *estimate*, not a verdict, and (for non-human results) points to
the appeal path.

### High-confidence AI  (`ai_probability` ≥ 0.75)

> ⚠️ **This content shows strong signals of AI generation** (confidence: High).
> Our automated analysis suggests it may not be human-written. This is an
> estimate, not a verdict — the creator can appeal this assessment.

### High-confidence human  (`ai_probability` < 0.40)

> ✓ **This content reads as human-written** (confidence: High). Our analysis
> found no strong signals of AI generation.

### Uncertain  (0.40 ≤ `ai_probability` < 0.75)

> ❓ **We couldn't confidently determine how this content was created**
> (confidence: Low). The signals are mixed. We're showing this transparently
> rather than guessing.

## 4. Appeals Workflow

- **Who can appeal:** the creator of any classified content (referenced by
  `content_id`). No automated re-classification is performed.
- **What they provide:** `content_id` + `creator_reasoning` (free text — e.g.
  "I wrote this myself; I'm a non-native speaker and write formally").
- **What the system does on appeal:**
  1. Look up the original decision by `content_id` (404 if unknown).
  2. Update that content's `status` → `under_review`.
  3. Append an appeal record to the audit log containing: the original
     attribution, confidence, both signal scores, the creator's reasoning, and
     timestamp.
  4. Return a confirmation `{content_id, status: "under_review", message}`.
- **What a human reviewer sees** (via `GET /log`, or a future review queue): the
  original classification, both individual signal scores and metrics, the
  combined confidence, and the creator's stated reasoning — enough context to
  make a human judgment.

## 5. Anticipated Edge Cases

1. **Formal / academic / non-native-English human writing.** Even, well-
   structured prose with low sentence-length variance scores AI-like on
   stylometry, and the LLM may also flag its "tidy" tone. Mitigation: the wide
   Uncertain band and high 0.75 AI threshold keep these in *Uncertain* rather
   than *Likely AI*, and the appeal path exists precisely for this case.

2. **Very short submissions (1–2 sentences) or poems with heavy repetition and
   simple vocabulary.** Stylometry is statistically unstable on short text —
   variance and TTR become unreliable, and a repetitive poem ("so much depends /
   upon...") can look uniform = AI-like. Mitigation: short-text inputs should
   lean toward *Uncertain*; we document this as a known limitation rather than
   pretending confidence we don't have.

3. **Lightly edited AI output.** A human who runs AI text through small edits can
   defeat both signals — meaning still reads coherent (LLM) and a few human edits
   raise variance (stylometry). This is fundamentally hard and acknowledged as a
   limitation; the honest answer is *Uncertain*, not a false verdict.

How each implementation milestone uses an AI tool, what spec sections feed it,
and how output is verified.

### M3 — Submission endpoint + Signal 1 (Groq LLM)

- **Spec provided:** "Detection Signals" (Signal 1 subsection) + the Architecture
  diagram + the API surface.
- **Ask the AI to generate:** the Flask app skeleton with a `POST /submit` route
  stub, the Signal-1 Groq function returning `{ai_probability, rationale}`, the
  SQLite audit-log helper, and a `GET /log` endpoint.
- **Verify:** call the Groq function directly on 2–3 sample inputs and inspect the
  raw output *before* wiring into the route; confirm the function signature
  matches the spec (returns a 0–1 probability, not a bare label); confirm
  `/submit` returns `content_id` + structured fields and writes a structured log
  entry.

### M4 — Signal 2 + confidence scoring

- **Spec provided:** "Detection Signals" (Signal 2 + Combination) + "Uncertainty
  Representation" (thresholds) + the diagram.
- **Ask the AI to generate:** the stylometry function returning
  `{ai_probability, metrics}`, and the scoring function combining both signals
  (`0.65*LLM + 0.35*stylometry`) and mapping to the three buckets.
- **Verify:** check the generated thresholds exactly match the table (0.40 /
  0.75) — AI tools often drift to "reasonable-looking" cutoffs; correct if so.
  Run the 4 calibration inputs and confirm scores match intuition; print both
  signal scores separately to locate any misbehaving signal.

### M5 — Production layer (label, appeal, rate limit, full log)

- **Spec provided:** "Transparency Label Design" (3 variants) + "Appeals
  Workflow" + the diagram.
- **Ask the AI to generate:** a label-generation function mapping score → exact
  label text, and the `POST /appeal` endpoint (status update + log).
- **Verify:** ask the AI to emit all three label variants and confirm the text
  matches the spec verbatim; submit inputs that hit each of the three bands;
  confirm `/appeal` flips status to `under_review` and logs the appeal; run the
  12-request curl loop and confirm 429s after the limit.

## Stretch Features

All four planned; `planning.md` will be updated with specifics before each is
built.

1. **Ensemble detection (3+ signals).** ✅ **BUILT.** Adds a third signal —
   **Signal 3: Predictability** (`signals/predictability.py`) — a pure-Python
   proxy for the low "perplexity" of AI text, combining two sub-metrics:
   - **Compression ratio** (`zlib`): `len(compressed)/len(original)`. More
     predictable/repetitive text compresses better → lower ratio → more AI-like.
   - **Distinct n-gram ratio** (bigram + trigram diversity): repetitive phrasing
     yields fewer distinct n-grams → more AI-like.
   Both capture information redundancy that neither the semantic LLM nor the
   surface stylometric metrics directly measure (compression-based detection is
   an established technique). Same short-text damping as stylometry.

   **Weighted vote** (in `scoring.py`): when all three signals are present,
   ```
   combined = 0.50*LLM + 0.30*stylometry + 0.20*predictability
   ```
   The LLM stays dominant; predictability carries the least weight (it is the
   noisiest) but breaks ties when the other two disagree. The 2-signal weighting
   (`0.65*LLM + 0.35*stylometry`) is retained as a fallback when only two signals
   are available. Thresholds (0.40 / 0.75) are unchanged.
2. **Provenance certificate.** ✅ **BUILT.** A "Verified Human" credential a
   creator earns through an extra verification step.
   - **`POST /verify-human`** `{creator_id, sample_text}` — the creator submits a
     live writing sample (minimum length enforced so the signals are
     meaningful). The full 3-signal pipeline runs on the sample; if it scores
     `likely_human` (high-confidence human), a certificate is issued, otherwise
     the request is rejected with the reason.
   - **Tamper-evident credential:** the certificate carries an HMAC-SHA256
     signature over `creator_id|certificate_id|issued_at` using a server secret
     (`PROVENANCE_SECRET`, stdlib `hmac`). `GET /certificate/<creator_id>`
     recomputes the HMAC and returns a `valid` flag, so a tampered credential is
     detectable.
   - **Display:** when a creator holding a valid certificate submits content,
     `/submit` returns `verified_human: true` and the transparency label is
     prefixed with `🏅 Verified Human Creator — `.
   - **Storage / privacy:** a `certificates` table stores
     `creator_id, certificate_id, issued_at, signature, status` — never the raw
     verification sample text.
3. **Analytics dashboard.** ✅ **BUILT.** Aggregates from SQLite (`analytics.py`)
   and is surfaced two ways:
   - **`GET /analytics`** — the aggregated metrics as JSON (canonical data).
   - **`GET /dashboard`** — a lightweight, self-contained HTML page (no external
     libraries; inline CSS + simple bars) rendering the same metrics.

   Metrics shown:
   - **Detection patterns:** count and percentage per attribution bucket
     (`likely_ai` / `uncertain` / `likely_human`) plus total classifications.
   - **Appeal rate:** appeals ÷ classifications (%).
   - **Additional metrics (chosen):** average confidence score (overall and per
     bucket) and the **verification issuance rate** (certificates issued vs
     rejected) — connecting the dashboard to the provenance-certificate feature.
4. **Multi-modal support.** 📋 **PLANNED — deferred to Phase 2 (product build).**
   Design below; not yet implemented. `/submit` will accept an optional
   `content_type` field (default `"text"` → the existing 3-signal pipeline,
   unchanged). A second type, **`image_metadata`**, carries a `metadata` dict and
   routes through a tailored provenance signal (`signals/metadata.py`):
   - **C2PA / Content Credentials** — `digital_source_type:
     "trainedAlgorithmicMedia"` or an explicit `ai_generated: true` assertion →
     strong AI (~0.95).
   - **Generator tool fingerprints** — `Midjourney`, `DALL·E`, `Stable
     Diffusion`, `Adobe Firefly`, `Imagen`, etc. found in any field → ~0.9.
   - **Camera/device EXIF present, no AI markers** → leans human (~0.15).
   - **No markers at all** (stripped metadata) → `0.5` uncertain (metadata is
     easily removed, so we stay honest).

   Rationale: real-world image provenance lives in metadata today (the C2PA
   standard), so this is the authentic signal a platform would check — and it is
   pure-Python with no API calls. The `image_metadata` path reuses the same
   scoring buckets, label generator, audit log, and certificate badge, so the
   user-facing contract (attribution / confidence / label / appeal) is identical
   across modalities. The audit log records `content_type`.
