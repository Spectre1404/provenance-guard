# Provenance Guard

A backend system any creative-sharing platform can plug into to assess whether
submitted **text** was written by a human or generated by AI — and, crucially,
to communicate **uncertainty honestly** rather than forcing a binary verdict.

Provenance Guard classifies content with multiple independent signals, produces
a calibrated confidence score, surfaces a plain-language transparency label,
lets creators appeal a classification, and records every decision in a
structured audit log behind rate limiting.

> **Design principle:** on a writing platform, a false positive (calling a
> human's work AI) is worse than a false negative. The scoring thresholds and
> label copy are deliberately biased to avoid wrongly accusing human writers.

## Architecture overview — the path a submission takes

```
Creator → POST /submit {text, creator_id}
   │  (rate limiter: 429 if over limit)
   ▼
 ┌── Signal 1: Groq LLM ──┐   ┌── Signal 2: Stylometry ──┐   ┌─ Signal 3: Predictability ─┐
 │   semantic, ai_prob    │   │  structural, ai_prob     │   │  redundancy, ai_prob/abstain │
 └───────────┬────────────┘   └────────────┬─────────────┘   └──────────────┬──────────────┘
             └───────────────── weighted vote ──────────────────────────────┘
                                     ▼
                       Confidence Scorer → combined ai_probability (0-1)
                                     ▼            → attribution bucket + confidence level
                       Label Generator → 1 of 3 transparency labels
                                     ▼  (+ "Verified Human" badge if creator is certified)
                       Audit Log (SQLite) ──────▶ Response JSON {content_id, attribution,
                                                  confidence, label, signals}
```

A submission first passes the **rate limiter**. The text is analyzed
independently by three signals (semantic, structural, redundancy). Their
`ai_probability` scores are combined into one **calibrated confidence**, mapped
to an attribution bucket (`likely_human` / `uncertain` / `likely_ai`) and a
user-facing confidence level, which drives the **transparency label**. The
decision is written to the **audit log** and returned with a unique
`content_id`. If the creator later disputes the result, `POST /appeal`
references that `content_id`, flips the content's status to `under_review`, and
logs the appeal alongside the original decision for a human reviewer.

### API surface

| Method & path | Purpose |
|---|---|
| `POST /submit` | Classify content, return attribution + confidence + label |
| `POST /appeal` | Contest a classification (status → `under_review`) |
| `GET  /log` | Recent structured audit-log entries |
| `POST /verify-human` | Earn a "Verified Human" provenance certificate *(stretch)* |
| `GET  /certificate/<creator_id>` | Fetch + validate a creator's certificate *(stretch)* |
| `GET  /analytics` | Aggregated metrics as JSON *(stretch)* |
| `GET  /dashboard` | HTML analytics view *(stretch)* |

### Code structure

```
app.py              Flask app + all routes
db.py               SQLite store + structured audit log
scoring.py          Signal combination + bucket/confidence mapping
labels.py           Transparency label generation
certificates.py     HMAC-signed "Verified Human" credential (stretch)
analytics.py        Dashboard aggregation (stretch)
signals/
  llm.py            Signal 1 — Groq LLM (semantic)
  stylometry.py     Signal 2 — stylometric heuristics (structural)
  predictability.py Signal 3 — information redundancy (stretch ensemble)
```

## Detection signals

The pipeline uses **three independent signals** (two are required; the third is
the ensemble stretch feature). "Independent" means they measure different
*kinds* of property, so agreement is more informative than any one alone.

### Signal 1 — Groq LLM classification (semantic)

- **Measures:** whether the text *reads* as machine-written when judged
  holistically — tonal evenness, generic phrasing, uniform hedging, tidy
  structure. Implemented by prompting `llama-3.3-70b-versatile` (temperature 0,
  JSON-only response) to return a probability, not a yes/no, and to lean
  mid-range when unsure.
- **Why chosen:** it is the only signal that understands *meaning*. LLM prose
  tends to be fluent and "safe"; human writing carries idiosyncrasy and opinion.
- **What it misses:** non-deterministic and sometimes confidently wrong; can be
  fooled by lightly edited AI text; **biased against formal or non-native
  English writers** whose careful prose resembles AI output. It never decides
  alone — it is weighted, and the "AI" threshold is deliberately high.

### Signal 2 — Stylometric heuristics (structural)

Pure Python, no external libraries. Six metrics across three families,
each normalized to a 0–1 AI-likeness sub-score, then weighted-averaged:

| Metric | Captures | Direction | Weight |
|---|---|---|---|
| Sentence-length burstiness (CV) | rhythm | low → AI | 0.22 |
| Type-token ratio | vocabulary diversity | mid-range → AI | 0.12 |
| Punctuation variety (distinct mark types) | richness | low → AI | 0.12 |
| AI-marker lexical density (*furthermore, delve, crucial, "it is important to note"…*) | AI register | high → AI | 0.24 |
| Human-voice markers (contractions, *honestly, ok so, kinda…*) | casual voice | high → human | 0.20 |
| Sentence-opener diversity | formulaic openings | low → AI | 0.10 |

- **Why chosen:** structural, language-statistics signal that is fully local and
  catches the *uniformity* of AI text the LLM might rationalize away.
- **What it misses:** **blind to meaning**; unstable on short text (< 3
  sentences → damped toward 0.5); a deliberately uniform human (legal, academic)
  scores AI-like; the AI-marker list can be gamed by avoiding those words. It
  carries the lower weight (0.35 in the 2-signal case) and never decides alone.

### Signal 3 — Predictability / information redundancy *(stretch: ensemble)*

- **Measures:** a pure-Python proxy for the low "perplexity" of AI text, via
  zlib compression ratio + distinct n-gram (bigram/trigram) ratio. More
  predictable/repetitive text compresses better and reuses more n-grams.
- **Abstention:** below ~100 words, AI and human prose are statistically
  indistinguishable on these metrics, so the signal marks itself `active:false`
  and the scorer drops it (renormalizing to the two-signal weighting) rather
  than letting a no-information reading dilute a confident verdict.
- **Privacy:** computed entirely locally — unlike the LLM signal it transmits no
  text anywhere, adding detection power with zero extra privacy exposure.
- **What it misses:** only meaningful on longer text; easily defeated by
  paraphrasing.

## Confidence scoring

Both/all signals output an `ai_probability` ∈ [0,1]. They are combined by a
**weighted average**:

- **Two signals** (predictability abstaining): `0.65·LLM + 0.35·stylometry`
- **Three signals** (predictability active): `0.50·LLM + 0.30·stylometry + 0.20·predictability`

The LLM carries the most weight (strongest semantic signal); predictability the
least (noisiest). The combined score maps to three buckets:

| Combined `ai_probability` | Attribution | User-facing confidence |
|---|---|---|
| `0.00 – 0.40` | `likely_human` | **High** |
| `0.40 – 0.75` | `uncertain` | **Low** |
| `0.75 – 1.00` | `likely_ai` | **High** |

**The asymmetry is the point.** A false positive is worse than a false negative,
so the "AI" bucket requires a high **0.75** to trigger while "human" only needs
< 0.40, and the *Uncertain* band is wide on purpose. User-facing confidence
reflects **distance from the uncertain middle**, not the raw score — which is
why a 0.51 (Uncertain/Low) feels very different to a reader than a 0.95
(Likely AI/High), exactly as the brief requires.

### How I validated the scores are meaningful

I ran four deliberately chosen inputs (clearly-AI, clearly-human, formal-human
borderline, lightly-edited-AI borderline) through the full pipeline and printed
each signal separately. The two clear cases separated widely (0.77 vs 0.18) and
**both borderline cases landed in the Uncertain band** — i.e. the system
declines to accuse when signals are mixed, instead of flipping at 0.5.

### Two example submissions (actual output)

**High-confidence example — `likely_ai`, confidence 0.7651**

Input: *"Artificial intelligence represents a transformative paradigm shift in
modern society. It is important to note that while the benefits of AI are
numerous, it is equally essential to consider the ethical implications.
Furthermore, stakeholders across various sectors must collaborate…"*

```json
{
  "attribution": "likely_ai",
  "confidence": 0.7651,
  "confidence_level": "High",
  "signals": {
    "llm":         { "ai_probability": 0.80 },
    "stylometry":  { "ai_probability": 0.7003 },
    "predictability": { "ai_probability": 0.474, "active": false }

}
```

**Lower-confidence example — `likely_human`, confidence 0.1822**

Input: *"ok so i finally tried that new ramen place downtown and honestly?


## Transparency label — three variants

The label returned by `/submit` changes with the confidence band. Exact text:

### High-confidence AI (`ai_probability` ≥ 0.75)

> ⚠️ This content shows strong signals of AI generation (confidence: High). Our
> automated analysis suggests it may not be human-written. This is an estimate,
> not a verdict — the creator can appeal this assessment.

### High-confidence human (`ai_probability` < 0.40)

> ✓ This content reads as human-written (confidence: High). Our analysis found
> no strong signals of AI generation.

### Uncertain (0.40 ≤ `ai_probability` < 0.75)

> ❓ We couldn't confidently determine how this content was created (confidence:
> Low). The signals are mixed. We're showing this transparently rather than
> guessing.

Every variant states it is an *estimate, not a verdict*, and the two non-human
variants point to the appeal path — reinforcing the false-positive-averse
design. A creator holding a **Verified Human** certificate (see stretch
features) gets the prefix `🏅 Verified Human Creator — ` before the label text.

## Appeals workflow

`POST /appeal` with `{content_id, creator_reasoning}`:

1. Looks up the original decision by `content_id` (404 if unknown).
2. Updates the content's status to `under_review`.
3. Appends an **appeal** event to the audit log containing the creator's
   reasoning *and* the original attribution + confidence, so a human reviewer
   has full context.
4. Returns `{content_id, status: "under_review", message}`.

Automated re-classification is intentionally not performed — a human reviews.

## Rate limiting

Applied to `/submit` via Flask-Limiter (keyed by client IP, in-memory storage):

```python
@limiter.limit("10 per minute;100 per day")
```

| Limit | Reasoning |
|---|---|
| **10 / minute** | A genuine writer submits work occasionally and may revise/resubmit a few times — 10/min is generous for that, while stopping a script from hammering the (paid) Groq backend in a tight loop. |
| **100 / day** | Caps daily cost/abuse from a single IP while comfortably covering any real creator's day of editing and resubmitting. |

### Evidence (12 rapid requests, limit 10/min)

```
Request#  Status
   1–10     400
   11       429   ← rate limit triggered
   12       429
```

> The requests above used minimal bodies so the demonstration would not consume
> ten Groq API calls. Flask-Limiter runs **before** the route handler, so each
> request still counts against the limit — the `400`s are just the empty test
> bodies being rejected by validation, and the switch to **`429` at request 11**
> is the rate limit firing. With valid bodies the first ten return `200`.

## Known limitations

- **Formal / academic / non-native-English human writing.** This is the system's
  most important failure mode and a direct consequence of its signals: such
  prose has low sentence-length variance and a "tidy" tone, so **both** the LLM
  (semantic) and stylometry (structural) push it upward. The formal-human
  example in the audit log above scored 0.5682 — correctly held in *Uncertain*
  rather than *Likely AI*, but a slightly higher score could misclassify a real
  human. The wide Uncertain band and the appeal path exist precisely for this.
- **Very short texts and repetitive poems.** Below ~3 sentences, stylometry is
  damped and predictability abstains entirely, so short submissions lean toward
  *Uncertain* — honest, but not confident.
- **Lightly edited AI output.** A few human edits can raise burstiness and keep
  meaning coherent, defeating all three signals. The honest output is
  *Uncertain*, not a false verdict.

## Spec reflection

- **How the spec helped:** writing the exact thresholds (0.40 / 0.75) and the
  three label texts *before* coding meant the scoring function had a concrete
  target. There was never a moment of "what should 0.6 mean?" mid-implementation
  — the answer ("Uncertain, not an accusation") was already decided, so the
  scorer produced graded labels instead of a binary flip at 0.5.
- **Where implementation diverged:** the planned third (ensemble) signal was a
  naive "proportion of repeated n-grams." When built and tested, it scored
  *every* short text as ~0.03 (because short prose is uniformly varied), which
  dragged a clearly-AI sample from 0.77 down to 0.62 and misclassified it. I
  diverged to an **abstention-aware** design: the signal recuses itself on short
  text and the scorer renormalizes, and the metric mappings were re-centered so
  typical prose reads as neutral. The spec named the signal; the real data
  forced a more honest combination rule.

## AI usage

This project was built with AI assistance (Claude). Two specific instances:

1. **Stylometry signal — directed and extended.** I directed the AI to generate
   a stylometric signal but pushed it past the obvious three metrics
   (variance / TTR / punctuation) toward a more discriminating set. It produced
   the six-metric design including the **AI-marker** and **human-voice** lexical
   detectors and the sentence-opener-diversity metric. I kept the design and the
   weighting that leans on the lexical tells, which turned out to be the
   strongest discriminators in testing.

2. **Ensemble signal — overrode the AI's first implementation.** I asked the AI
   to add a third "predictability" signal and a weighted vote. Its first version
   used naive compression/n-gram reference ranges that scored everything as
   "human" and **broke clearly-AI detection** (pulled 0.77 → 0.62). I rejected
   that and directed a fix: length-based **abstention** plus recalibrated,
   centered mappings so the signal only votes when it has real information. The
   validation step that caught the regression was itself part of the spec's
   "verify the generated scoring matches your thresholds" instruction.
